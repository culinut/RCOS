# Project Background

## 项目是什么
`news_agent_mvp` 是一个以 Python / FastAPI 实现的新闻聚合与摘要 MVP。它对外提供 HTTP API 和浏览器聊天页，使用独立的 `mcp_news_server.py` 从 RSS 热榜源取数，再通过 LLM 完成：
- 是否需要新闻的前置判断；
- 对候选条目的相关性评分；
- 中文标题 / 中文摘要 / 相关标签生成；
- 最终摘要与后续建议生成。

从当前源码事实看，它更准确地属于 **单轮 agentic workflow**，而不是完整 autonomous agent：
- 有 LLM 参与的规划式判断；
- 有外部工具调用；
- 有多阶段处理链路；
- 但没有循环、记忆、反思或自动补救闭环。

## 为什么存在
这个项目的直接目标，是把“直接看原始热榜/RSS”升级为“先筛一遍、再给中文可消费摘要”。当前代码已经体现出几个明确意图：
- 不是单纯抓取，而是先判断用户是否真的在问近期新闻；
- 不是拿到条目就原样展示，而是先做相关性评分和中文本地化；
- 不是只输出一段摘要，而是同时给出排序后的候选项和后续追问建议。

换句话说，项目要解决的是：
- 原始信息流噪音高；
- 用户通常只关心与自己问题相关的少数条目；
- 英文 RSS 条目对中文用户不够直接；
- 团队需要一个可快速试验 prompt、评分逻辑和界面的最小闭环。

## 价值 / 目标用户
当前源码最能支持的目标用户判断有两类。

### 1. 团队内部试验用户
这是当前最明确的一类用户：
- 能直接通过 `/chat` 提问；
- 能通过 `/log`、`/log/raw` 看日志；
- 能通过 `/refresh` 重启服务；
- 可以迭代 prompt、源配置、评分与展示方式。

因此，它首先是一个方便团队快速试错和对齐的实验平台。

### 2. 希望快速获取技术/AI 热点摘要的用户
从现有数据源配置看，当前默认来源是：
- Hacker News Front Page
- Hacker News Newest
- Reddit OpenAI

所以它当前更适合“想快速了解技术 / AI 社区热点的人”，而不是泛新闻用户，也不是中文主流资讯用户。

## 当前范围边界

### 当前已在范围内
- 基于 RSS 的热榜获取；
- 基于用户问题判断是否需要看新闻；
- 单轮拉取一个 source 的候选条目；
- 相关性评分；
- 时效性评分与标签；
- 中文标题 / 中文摘要 / 相关标签生成；
- 最终摘要、重点条目、后续建议输出；
- 浏览器聊天页、日志页、健康检查与重启入口。

### 当前不在范围内
- 长期记忆 / RAG；
- Reflection / 自我修正闭环；
- 多 agent；
- 来源可信度评分；
- 正文级深度分析主流程；
- 通用搜索引擎式的新闻检索；
- 面向大规模生产环境的可靠性工程。

## 当前定位总结
当前仓库的定位更准确地说是：

> 一个面向技术 / AI 热榜场景的新闻摘要 MVP，用最小可运行闭环把“问题判断 → RSS 拉取 → 评分 → 中文本地化 → 摘要展示”串起来，方便团队继续验证更强的搜索、正文抓取和排序能力。
