# PROJECT_ROADMAP

## 目的

本文定义 `SanXing-AsItIs` 从当前基线推进到可用产品的建议路线图。

它不是功能承诺清单，而是：

- 当前推荐实现顺序
- 模块依赖关系的执行化表达
- 里程碑级交付物与验收标准

---

## 当前阶段判断

根据当前代码与 RCOS 项目文档，本项目处于：

👉 `M1` 后半段，尚未正式进入 `M2`

原因：

- 评分语义 seed 已存在，可作为工作基线
- 最小 FastAPI + systemd 运行骨架已存在
- 但 `checkin / storage / ui / export_pdf` 仍未落地
- SQLite 目前仅为空文件占位

---

## 总体路线

推荐按以下顺序推进：

1. `M1` 语义定型
2. `M2` 数据闭环
3. `M3` 交互闭环
4. `M4` 导出闭环

该顺序的核心原则是：

- 先稳定语义
- 再稳定数据
- 再稳定交互
- 最后处理表现层导出

---

## M1: 语义定型

### 目标

将 `scoring_model_seed.json` 从“工作语义基线”推进为“可直接编码的数据契约”。

### 交付物

- 稳定的时间切片定义：
  - `morning`
  - `midday`
  - `evening`
- 稳定的预读结构：
  - 六度
  - 八正道
- 稳定的 12 维评分结构
- 稳定的统一评分语义：
  - 高分 = 更符合修行方向
- 三省执行层与评分维度层的关系说明

### 主要模块

- `scoring_model`
- 部分 `eightfold_display` 规则定义

### 当前阻塞点

- seed 虽可用，但仍未完成逐字级 PDF 校勘
- `s1 / s2 / s3` 与 12 维的映射关系尚未正式定稿

### 验收标准

- 能明确前端需要展示哪些字段
- 能明确后端需要保存哪些字段
- 不再需要依赖口头解释评分逻辑
- 语义结构足以指导 schema 设计

---

## M2: 数据闭环

### 目标

把语义结构落成可保存、可读取、可验证的数据闭环。

### 交付物

- SQLite schema
- check-in 数据模型
- 最小持久化读写逻辑
- 最小 API 读写接口

### 主要模块

- `storage`
- `checkin`
- `api`

### 推荐最小数据范围

- `date`
- `period`
- `reviewed_eightfold`
- `s1`
- `s2`
- `s3`
- `optional_note`

如果 12 维需要直接落库，则还需进一步明确：

- 12 维是否作为主存储字段
- 还是作为计算层 / 解释层
- 还是与 `s1 / s2 / s3` 形成聚合关系

### 验收标准

- 能提交一次 check-in
- 能落库
- 能读取指定日期 / 时段的记录
- API 行为与语义模型一致

---

## M3: 交互闭环

### 目标

让用户可以在网页中完成：

预读 → 确认 → check-in → 保存完成

### 交付物

- 八正道预读页或预读区块
- 明确的“已阅读确认”
- 低摩擦 check-in 表单
- 提交完成反馈

### 主要模块

- `ui`
- `eightfold_display`
- `api`

### 设计原则

- 优先结构化输入
- 减少自由文本依赖
- 手机与桌面都可用
- gating 必须明确

### 验收标准

- 用户可在手机和桌面完成一次完整流程
- 未完成预读确认时不能直接提交
- 表单操作路径足够短

---

## M4: 导出闭环

### 目标

将已保存的 check-in 记录导出为模拟手填纸质《三省书》的 PDF。

### 交付物

- PDF 模板资源
- 数据到纸质版面的映射规则
- 导出文件输出到 `pdf/export/`

### 主要模块

- `export_pdf`
- `storage`

### 说明

`pdf/export/` 当前已被定义为导出结果目录。

`pdf/template/` 当前可视为后续模板资源目录。

### 验收标准

- 能选择记录并导出 PDF
- 导出结果与纸质样式保持大体一致
- 文件可保存、可查看、可复核

---

## 当前建议路线

从当前状态出发，建议按以下顺序推进：

1. 先把 `scoring_model` 收紧成稳定 schema
2. 明确 `12 维` 与 `s1 / s2 / s3` 的关系
3. 设计 SQLite schema
4. 提供最小 API
5. 实现最小 UI
6. 最后做 PDF export

---

## 当前关键未决策项

### 决策 1: `s1 / s2 / s3` 的含义

当前最关键的产品 / 数据建模问题是：

- `s1 / s2 / s3` 是三个总分槽位
- 还是三省反思项
- 还是三次时间片内的聚合结果

这个决策会直接影响：

- 数据库 schema
- API payload
- UI 表单结构
- 导出 PDF 的映射方式

### 决策 2: 12 维的落库方式

需要明确：

- 12 维是主存储对象
- 还是只作为评分解释层

### 决策 3: seed 的核对深度

当前 `seed` 已可用，但仍需明确后续是否要：

- 做 OCR 辅助核对
- 还是人工逐页校勘
- 还是保持“工作基线”状态继续推进实现

## 使用方式

当团队讨论“下一步做什么”时，优先参考本文。

当团队讨论“这个需求现在该不该做”时，优先检查：

- 它属于哪个里程碑
- 是否会绕过当前推荐顺序
- 是否会在语义未稳定前提前进入 UI 或导出实现

- 当前阶段判断变化
- 推荐实现顺序变化

- 某个里程碑已基本完成
